4. DATA FLOW

data_flow.md

1. User writes code
2. Editor Service saves file
3. Sync Service updates collaborators
4. User requests agent task
5. Task goes to Agent Manager
6. Task pushed to Message Queue
7. Agent picks task
8. Agent processes data
9. Result stored in database
10. Sync Service updates editor

6. DATABASE DESIGN

SQL Schema

sql_schema.sql

CREATE TABLE users (
  user_id SERIAL PRIMARY KEY,
  name TEXT,
  email TEXT,
  password TEXT,
  created_at TIMESTAMP
);

CREATE TABLE projects (
  project_id SERIAL PRIMARY KEY,
  user_id INT,
  project_name TEXT,
  created_at TIMESTAMP
);

CREATE TABLE files (
  file_id SERIAL PRIMARY KEY,
  project_id INT,
  file_name TEXT,
  file_content TEXT,
  updated_at TIMESTAMP
);

CREATE TABLE agent_tasks (
  task_id SERIAL PRIMARY KEY,
  project_id INT,
  agent_type TEXT,
  status TEXT,
  result TEXT,
  created_at TIMESTAMP
);

CREATE TABLE versions (
  version_id SERIAL PRIMARY KEY,
  file_id INT,
  content TEXT,
  created_at TIMESTAMP
);

7. DATA SYNCHRONIZATION MODEL

This is very important for your system.

Sync Service Uses:

WebSockets

CRDT

Operational Transform


Sync Flow

User edits file
      |
Editor Service
      |
Database
      |
Sync Service
      |
WebSocket Server
      |
All Connected Clients + Agents

This ensures all agents and users see same data.

8. MESSAGE QUEUE SYSTEM

Queue stores agent tasks.

Queue Message Example

{
  task_id: 101,
  agent: "documentation",
  project_id: 55,
  file_id: 22,
  input: "Generate documentation for this file"
}

Agents pick tasks asynchronously.
